The Problem
The ML pipeline existed but had five critical flaws:
1. Training data was random noise — generate_more_data.py created flowering DOY targets by picking random numbers, not by modeling actual plant biology. The model couldn't learn weather → flowering relationships because there were none in the data.
2. Missing scalers — train_model.py wrote code to save StandardScaler objects, but they were never actually saved. predict.py tried to load them, they didn't exist, so it fell back to the hardcoded baseline every time.
3. Backend model path was wrong — prediction_service.py looked for models at D:\Pollysync\models\ but models lived at D:\Pollysync\ml\models\. The error was silently swallowed by try/except Exception: pass.
4. No feature alignment — The backend's build_features() didn't include bee_count (only bee_richness), but the model expected both. Values passed to .predict() went in without scaling.
5. Models were undertuned — All three models used the same arbitrary hyperparameters (max_depth=8, n_estimators=200) with no tuning.

Phase 2 — Fix Training Data
This is the most impactful change. Understanding why the original data was bad requires understanding GDD (Growing Degree Days).
How GDD works
Plants need cumulative heat to flower. Each day, you compute:
daily_GDD = max( (T_max + T_min) / 2 - T_base, 0 )
For sunflowers: T_base = 6.7°C, GDD to flowering = 958 units. So at 25°C average daily temp, you get ~18.3 GDD/day, and flowering takes 958/18.3 ≈ 52 days from sowing.
What we changed
The old generate_more_data.py did this:
flowering_doy = base_flowering_doy + random_noise  # NO relationship to temp
The new version does this:
gdd_df = build_gdd_series(temp_df, crop)
hit = gdd_df[gdd_df["cumulative_gdd"] >= params["gdd_to_flowering"]]
flowering_date = hit.iloc[0]["date"]
start_doy = flowering_date.timetuple().tm_yday
Now higher temperature → faster GDD accumulation → earlier flowering DOY. The model can actually learn the physical relationship.
PSI data improvements
The old PSI scoring function was identical to the baseline rule-based scorer. The model was learning to approximate a rule it already knew. We added:
- Wind speed penalty (wind > 20 km/h reduces score)
- Rainfall sweet spot (2-10mm is good, >20mm is bad)
- NDVI × bee richness interaction term
- Bee abundance scaling factor
Results
After retraining with the new data:
- Feature importance: temp_7d_mean went from 0.006 to competitive with crop/month features
- R² on synthetic test: 0.9997 (near-perfect, as expected — the GDD model is deterministic)

Feature engineering (V2 models)
We added 7 interaction terms to capture non-linear relationships:
temp_humidity    = temp × humidity / 100     (heat index proxy)
temp_ndvi        = temp × ndvi               (vegetation health under heat)
humidity_rainfall = humidity × rainfall / 100 (muggy vs dry conditions)
temp_sq           = temp²                    (optimal temp range is non-linear)
ndvi_sq           = ndvi²                    (diminishing returns on greenness)
bee_total         = richness × abundance     (total pollinator population)
temp_wind         = temp × wind_speed        (wind chill effect on pollinators)
V2 models are saved as *_v2.pkl alongside the default V1 models.
